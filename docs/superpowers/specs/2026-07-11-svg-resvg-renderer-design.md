# SVG + resvg 渲染器重构设计

## 目标

用 SVG + resvg 替换生产环境中的 HTML + Headless Chromium 渲染链路，降低常驻内存和单次渲染开销，同时保持现有 HTTP 接口、请求格式、主题规则和消息展示能力不变。

视觉验收以现有 Chromium 输出为基准：画布尺寸、换行、间距、颜色、圆角、图片缩放和元素位置必须一致；允许文字边缘因栅格化引擎不同而产生极小的抗锯齿差异。

## 范围

重构覆盖：

- `/png/` 和 `/base64/` 背后的生产渲染器；
- 纯文本、图文混排、自定义头像、QQ 头像、emoji、sticker 和昼夜主题；
- HTTP、HTTPS 和 `data:image/*` 图片资源；
- Docker 构建、运行时字体和部署说明；
- 旧 Chromium 渲染器备份及双引擎视觉回归工具。

不包含：

- 新增或修改 HTTP 接口；
- HTTP 接口自动化测试；
- 性能基准测试；
- 当前功能以外的通用 HTML/CSS 布局能力。

## 技术路线

采用“Go 固定规则布局 + SVG 序列化 + resvg 原生库”的实现。

不使用 SVG `foreignObject`，因为 resvg 不提供浏览器式 HTML/CSS 排版，无法保证兼容性。不把通用 Canvas/SVG 布局库作为核心布局层，避免第三方换行和字体回退规则造成不可控偏差。

项目允许启用 CGO。Docker 构建阶段负责构建或链接 resvg 原生库，运行镜像不再安装 Chromium。生产依赖中移除 rod，删除浏览器池和 `POOL_SIZE` 配置；Dockerfile、README 和部署示例同步删除 `POOL_SIZE`。

## 架构

### Renderer

`Renderer` 继续提供：

- `Render(context.Context, []Message) ([]byte, error)`
- `RenderBase64(context.Context, []Message) (string, error)`

HTTP 层无需了解渲染实现。`RenderBase64` 继续对 `Render` 结果做标准 Base64 编码。

### ResourceLoader

资源加载器负责 HTTP、HTTPS 和 data URI：

- 每个请求受渲染上下文和明确超时控制；
- HTTP 响应体设置固定上限，防止异常资源耗尽内存；
- 成功资源解码为可测量的图像，并在 SVG 中以内嵌数据表示；
- 单张远程资源失败返回明确的缺失状态，不终止整张引用图渲染；
- SVG 交给 resvg 时不再包含外部网络引用。

资源加载器只报告真实加载结果，不猜测替代地址，不生成伪造图片。

### FontManager

字体管理器负责：

- 加载项目指定的中文无衬线字体；
- 提供与布局引擎一致的文字宽度和行高测量；
- 为中文、拉丁字符、常用符号及 emoji 选择固定字体；
- 保证本地视觉回归与 Docker 运行环境使用同一套字体文件。

字体缺失属于渲染初始化错误，必须明确失败，不使用不可控的系统字体兜底。

### LayoutEngine

布局引擎只实现现有模板实际使用的规则：

- 卡片最大宽度 600px、水平内边距 12px、垂直内边距 16px、圆角 12px；
- 消息间距 14px，头像与内容间距 10px；
- 42×42px 圆形头像；
- 12px 昵称、4px 昵称下间距、单行展示；
- 气泡四角半径为 4/12/12/12px，内边距为 8px 12px；
- 15px 正文、1.6 行高、保留换行和空白、按词内断行；
- 普通图片最大 200×200px，使用 contain 缩放并保留宽高比；
- emoji 固定 42×42px，sticker 最大 96×96px；
- 相邻消息段垂直间距 4px；
- 画布按内容自适应，但不得超过当前 600px 卡片上限。

布局引擎输出纯几何布局树，包括画布、消息、文本行、头像和消息图片的坐标与尺寸。它不拼接 SVG，也不下载资源。

### SVGBuilder

SVG 构建器把布局树序列化为完整 SVG：

- 主题背景和气泡使用 SVG 基础图形；
- 头像通过圆形 clipPath 裁剪；
- 图片使用 preserveAspectRatio 表达 contain 行为；
- 文本逐行定位，用户文本和昵称严格进行 XML 转义；
- 图片只使用资源加载器提供的内嵌数据；
- 输出尺寸取布局树的实际画布尺寸。

### ResvgRenderer

resvg 适配器通过 CGO 调用原生库，将内存中的 SVG 渲染为 PNG。它只负责格式转换，不参与布局、资源加载或错误降级。

### Legacy Chromium Renderer

现有 Chromium 实现完整迁移到 `legacy/chromium`，保留模板、页面池及旧渲染器，只供视觉回归工具使用。旧实现不编入生产服务，也不进入生产运行镜像。

### Visual Regression Tool

`cmd/visual-regression` 接收固定 JSON 和本地素材，用相同消息、固定主题时间和字体分别调用旧、新渲染器，输出：

- Chromium 基准 PNG；
- SVG + resvg PNG；
- 像素差异图；
- 包含尺寸和差异统计的 JSON 报告。

正式回归夹具全部使用本地素材，避免网络状态污染结果。另以网上公开素材做一次人工下载链路验证，但不把外部网络作为自动回归的前置条件。

## 数据流

1. Gin 按现有方式将 JSON 绑定到 `[]Message`；非法 JSON 继续返回 400。
2. 消息预处理保持现有语义：字符串变成文本段，数组解析为图文段，其他 JSON 值按当前格式化行为变成文本。
3. 资源加载器解析并加载头像、普通图片、emoji 和 sticker，记录成功或缺失状态。
4. 字体管理器测量文字，布局引擎计算所有元素和最终画布尺寸。
5. SVG 构建器生成仅含内嵌资源的 SVG。
6. resvg 将 SVG 渲染为 PNG。
7. `/png/` 返回 `image/png`；`/base64/` 返回 PNG 的纯 Base64 文本。

主题规则保持不变：服务器本地时间 06:00–17:59 使用浅色主题，其余时间使用深色主题。视觉回归工具允许显式固定主题，避免测试随运行时间变化。

## 错误处理

- 非法 JSON：HTTP 400，保持现有行为。
- 头像加载失败：继续渲染主题对应的圆形头像底色。
- 消息图片加载失败：保留该图片段的布局语义，但不伪造内容，继续生成整张图。
- 字体加载、SVG 构建、图片解码或 resvg 渲染失败：返回 HTTP 500，并记录失败阶段。
- 请求取消或超时：终止仍在进行的资源加载和渲染，返回真实错误。

远程资源失败的兼容目标是与当前 Chromium 一样继续出图，而不是把单张资源失败升级为整次请求失败。

## 测试与验收

### 代码级测试

单元测试覆盖：

- 原有消息字段兼容规则；
- 昼夜主题边界；
- URL 和 data URI 校验；
- 图片 contain 缩放；
- 普通图片、emoji 和 sticker 尺寸；
- 文本保留换行、长文本断行和空白处理；
- 昵称、气泡、头像、消息间距和画布尺寸；
- SVG XML 转义、圆形裁剪和内嵌图片；
- 远程头像及消息图片失败时继续生成布局。

不新增 HTTP 接口测试；接口兼容通过保留现有路由、请求类型、Renderer 方法和响应写法，在代码层面保证。

### 视觉回归

固定样例至少包含：

- 中英文短文本；
- 长中文、长英文和显式换行；
- 多条不同昵称消息；
- 方形头像和透明头像；
- 横图、竖图和透明 PNG；
- 文字与图片混排；
- emoji 和 sticker；
- 资源加载失败；
- 浅色和深色主题。

自动比较要求：

- 输出宽高完全相同；
- 卡片、头像、昵称、气泡、图片的几何边界一致；
- 非文字区域逐像素一致；
- 文字区域允许极小抗锯齿差异，并输出差异比例供人工复核；
- 每个固定样例保存叠加图或热力差异图。

## 部署变化

- 生产 Go 依赖移除 rod；
- 删除 `pool.go` 和生产浏览器初始化；
- 删除 `POOL_SIZE` 环境变量及全部部署示例；
- Docker 构建阶段启用 CGO 并构建/链接 resvg；
- 运行镜像只保留应用、resvg 所需动态库及固定字体，不安装 Chromium；
- `PORT` 默认值及监听行为不变。

## 完成条件

- 生产服务不启动或依赖 Chromium；
- 所有原有消息格式和渲染特性由 SVG 路径支持；
- `/png/`、`/base64/` 的代码级契约保持不变；
- `POOL_SIZE` 从生产代码、Dockerfile、README 和部署示例中删除；
- 单元测试全部通过；
- 双引擎固定样例的画布和几何布局一致，差异仅限允许的文字抗锯齿范围；
- 旧 Chromium 实现可独立运行视觉回归，但不进入生产产物。
